# Coding Guidelines

## 1. Scope проекта

- Репозиторий ведется как **backend-only** (NestJS + TypeScript).
- Фронтенд-правила (React, hooks, UI-паттерны) в этом документе не применяются.
- Любые архитектурные решения принимаются с приоритетом на API, доменную логику, интеграции и стабильность продакшена.

---

## 2. Базовые принципы

- Читаемость и предсказуемость важнее «краткого» кода.
- Один файл/класс/функция должны иметь одну ответственность.
- Не использовать `any`; если точный тип пока невозможен, использовать `unknown` и сужение типа.
- «Магические» числа и строки выносить в именованные константы или `enum`.
- Комментарии писать только там, где нужно объяснить причину решения, а не очевидное поведение кода.

---

## 3. Структура backend-кода

- Код группируется по доменам в `src` (например: `auth`, `users`, `tiles`, `players`).
- Для каждого домена сохранять слойность Nest:
  - `*.module.ts` — сборка зависимостей.
  - `*.controller.ts` — HTTP-слой и контракт API.
  - `*.service.ts` — бизнес-логика.
  - `dto/` — входные и выходные DTO.
- Общий код, который реально переиспользуется, размещать в `packages/shared`.
- Не создавать абстракции «на вырост» без реального второго сценария использования.

---

## 4. Импорты и алиасы

- Внутри `src` использовать абсолютные импорты через `@/*`.
- Общие сущности из shared-пакета импортировать через `@shared/*`.
- Не использовать глубокие относительные цепочки вида `../../../`.
- Импорты типов отделять от value-импортов через `import type`.

---

## 5. TypeScript и линтинг

- TypeScript работает в строгом режиме; не ослаблять настройки `tsconfig` без причины.
- Перед коммитом обязательно проходить:
  - `npm run lint`
  - `npm run test`
  - при изменении контрактов/DTO: `npm run build`
- Форматирование и стиль кода контролируются ESLint + Prettier и являются обязательными.

---

## 6. API и NestJS-практики

- Контроллеры должны быть тонкими: в них только маршрутизация, валидация и маппинг.
- Бизнес-правила и проверки размещать в сервисах/доменном слое.
- Все входные данные для API описывать DTO и валидировать (`class-validator`).
- Не выбрасывать «сырые» ошибки наружу; использовать HTTP-исключения Nest с понятными сообщениями.
- Изменения API-контрактов должны быть явными и обратно совместимыми, если не согласован breaking change.

---

## 7. Работа с данными и конфигом

- Значения из окружения валидировать на старте приложения (fail-fast).
- Не хардкодить секреты, URI и порты в коде.
- В запросах к БД избегать неявной логики и скрытых сайд-эффектов.
- Для транзакционных и критичных операций явно описывать порядок шагов и обработку ошибок.

---

## 8. Тестирование

- Unit-тесты обязательны для сервисов с нетривиальной бизнес-логикой.
- Интеграционные/e2e-тесты обязательны для критичных API-потоков (auth, записи в БД, права доступа).
- Каждый багфикс должен сопровождаться тестом, который воспроизводит проблему.
- Тесты должны быть детерминированными и изолированными.

---

## 9. Git и изменения

- Использовать [Conventional Commits](https://www.conventionalcommits.org/):
  - `feat: add refresh token rotation`
  - `fix: validate DATABASE_PORT on bootstrap`
  - `refactor: split auth service responsibilities`
- Один PR = одна логическая задача.
- В описании PR указывать:
  - что изменено,
  - зачем изменено,
  - какие риски и как проверено.
